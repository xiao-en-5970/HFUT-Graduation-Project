# 订单与聊天（平台不经手资金）

本系统**不处理任何资金**，买卖双方线下沟通、扫码付款。

## 商品类别 `goods_type`（商品表）

| 值 | 含义             | 订单履约                                                                                                                         |
|---|----------------|------------------------------------------------------------------------------------------------------------------------------|
| 1 | **送货上门**       | 双方同意后 → **正在派送** → 卖方 `confirm-delivery` → 待买方确认收货；买卖地址齐全时可算步行距离                                                             |
| 2 | **自提**         | 卖方在商品上填 `pickup_addr`（约定提货点）。买家下单时若未填 `receiver_addr`，默认用 `pickup_addr` 作为收货/提货说明；**不计算**送货距离；仍走：同意 → 正在派送 → 卖方确认送达 → 买方确认收货 |
| 3 | **在线商品**（会员卡等） | **无实体派送**；双方同意后**直接进入「待买方确认收货」**（视为卖方已发货），**不再调用** `confirm-delivery`；买方确认收货后扣库存                                             |

## 订单状态 `order_status`

| 值 | 含义                                   |
|---|--------------------------------------|
| 1 | **待下单**：买家「我想要」后，双方可聊天               |
| 2 | **正在派送**：送货上门/自提：双方同意后进入；在线商品不会进入此状态 |
| 3 | **待买方确认收货**：卖方已送达（实体）或在线商品双方已同意      |
| 4 | **已完成**：买方确认收货，**库存 -1**             |
| 5 | **已取消**：待下单或正在派送阶段可取消                |

## 典型 API 顺序（前端）

### 送货上门 / 自提

1. `POST /api/v1/orders` — `{ "goods_id", "receiver_addr"?: "可选；自提可省略，将用商品 pickup_addr" }`
2. `GET/POST /api/v1/orders/:id/messages` — 聊天
3. `POST /api/v1/orders/:id/agree` — 买卖双方各一次 → **正在派送**（状态 2）
4. `PUT /api/v1/orders/:id` — 卖方可填 `sender_addr`（待下单或正在派送）
5. `POST /api/v1/orders/:id/confirm-delivery` — 卖方，可选 `delivery_images` → 状态 3
6. `POST /api/v1/orders/:id/confirm-receipt` — 买方 → 状态 4

### 在线商品

1～3 同上；第 3 步双方同意后**直接为状态 3**，跳过 `confirm-delivery`。

4. `POST /api/v1/orders/:id/confirm-receipt` — 买方 → 状态 4

图片请先走 `POST /api/v1/oss/...` 上传，再将返回 URL 写入消息或确认接口。

## 管理后台演示（两账号模拟交易）

1. 在「商品管理」创建商品并选择 **类别**（送货上门/自提/在线），自提类填写 **自提地址**。
2. 两用户绑定同一学校，分别模拟买家/卖家完成交易。
3. 「订单演示」可查看订单与聊天记录。

数据库变更：`migrate_order_chat.sql`、`migrate_goods_type.sql`。
